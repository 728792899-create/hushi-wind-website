# 安全策略

[English](SECURITY.md) · **简体中文**

## 支持版本

安全修复应用于默认分支的最新提交。Demo 凭据、fixture 数据和本地 SQLite 文件不属于生产凭据或生产数据。

## 报告漏洞

不要创建包含利用细节、凭据或客户数据的公开 Issue。请通过 GitHub 私密 Security Advisory 联系仓库所有者，并提供：

- 受影响的路由、组件和提交；
- 影响与最小复现；
- 是否需要鉴权或特定权限；
- 已脱敏的请求/响应证据；
- 已知的缓解建议。

预期在三个工作日内确认。鉴权绕过、任意文件上传、远程代码执行、密钥暴露和客户数据泄露优先处理；公开披露应等待修复与部署窗口达成一致。

## 安全假设

- 生产运行在 HTTPS 和可信反向代理之后，只有代理链可控时才设置 `TRUST_PROXY`。
- 密钥由部署平台注入；`.env`、数据库、备份和运行时上传不进入仓库。
- 多实例生产使用 PostgreSQL、Redis 共享限流和对象存储；SQLite/本地文件仅用于 Demo 或受控单实例。
- Sentry 与产品分析默认不采集 PII，禁止加入咨询正文、联系方式、token 或 cookie。
- 疑似泄露后轮换管理员密码、token secret、S3、Redis 与 Sentry 构建凭据。

## 运营响应

1. 禁用或轮换受影响凭据/会话。
2. 保留结构化日志和审计记录，不把客户 PII 复制到工单。
3. 当隔离风险低于热修时回滚受影响 release。
4. 只从已验证备份恢复，并执行发布后冒烟。
5. 记录根因、影响窗口和纠正测试。
