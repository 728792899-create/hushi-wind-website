# 3 分钟本地三端演示与录屏脚本

这份脚本用于作品集、面试和代码审查演示。官网、CMS 后台和 API 全部在本机运行，不依赖已经下线或过期的历史服务器。

## 录制前准备

首次运行：

```bash
npm run install:all
npm run seed:demo
npm run dev
```

看到三个服务启动后，分别打开：

- 官网：`http://127.0.0.1:3000`
- CMS 后台：`http://127.0.0.1:5175`
- API 健康检查：`http://127.0.0.1:1337/health`

后台演示账号仅用于本地：

```text
demo_admin / DemoPass_2026!
```

录制前确认浏览器中没有真实客户资料、生产环境 Cookie、服务器地址、密钥或私人联系方式。建议使用独立浏览器窗口，并提前打开三个标签页。

## 三分钟时间线

### 00:00–00:20｜问题与项目边界

画面：官网首页。

讲解：

> 这是一个面向乐器品牌持续运营场景的三端项目。纯静态网站无法让运营人员独立维护产品、文章和咨询内容，所以我把系统拆成 Nuxt 官网、Vue CMS 后台和 Express API，并提供完整虚构内容的一键本地演示。

### 00:20–00:50｜官网用户路径

画面：依次展示产品列表、产品详情、文章或支持页面。

重点说明：

- 用户可以浏览结构化产品、文章和支持内容。
- 页面数据来自 API，而不是写死在页面组件中。
- 产品详情、SEO 字段和支持内容均可由后台维护。

### 00:50–01:35｜CMS 内容维护

画面：进入后台，用演示账号登录，打开产品或 FAQ 管理。

操作：

1. 编辑一条明确标记为 Demo 的产品标题或描述。
2. 保存后展示成功反馈。
3. 不删除数据，不展示管理员 Token、审计详情中的来源信息或浏览器存储。

讲解：

> 后台不是单独的静态原型，而是通过鉴权 API 修改同一套业务数据。高权限接口有登录、权限校验、输入校验和审计边界；公开仓库不会携带生产数据库和真实账号。

### 01:35–01:55｜前后台闭环

画面：返回官网并刷新对应页面，确认刚才修改的 Demo 内容已经更新。

讲解：

> 这一步证明前台展示、后台维护和 API 写入形成了真实闭环，也说明为什么没有采用成本更低但无法交接运营的纯静态站方案。

### 01:55–02:25｜API 与运行状态

画面：打开 `/health`，只展示状态、依赖检查和服务版本等非敏感字段；随后可在终端运行：

```bash
curl -I http://127.0.0.1:1337/health
```

重点说明：

- API 默认仅监听本机回环地址；使用弱演示凭据时不能监听全网卡。
- 持久日志中的 IP 会在写入前截断匿名化。
- 应用层限流是单实例兜底，生产多实例应由 Nginx 等统一网关限流。

### 02:25–02:50｜工程与安全取舍

画面：展示项目目录、`prisma/schema.prisma`、测试/验收脚本或终端中的质量检查结果。

讲解：

> 当前项目选择 Prisma 和 SQLite，是为了满足单机演示与小规模交付，而不是假装已经达到大型 SaaS 规模。上线检查覆盖官网、后台和 API，生产构建会拒绝 localhost 配置；公开 seed 使用虚构业务数据，演示凭据不会通过前台 FAQ 暴露。

### 02:50–03:00｜收尾

画面：回到官网首页或三端标签页总览。

讲解：

> 项目完成了内容展示、CMS 维护、API、安全校验和交付说明的完整闭环。历史线上服务器已经下线，因此作品集以 GitHub 源码、自动化检查和本地可复现演示作为当前证据。

## 录制完成检查

- 视频时长控制在 2 分 40 秒至 3 分 20 秒。
- 官网、后台、API 三端至少各出现一次。
- 至少展示一次“后台修改 → 官网刷新生效”的闭环。
- 不出现真实数据库、上传素材、生产域名后台、密钥、Cookie、Token 或客户信息。
- 不宣称历史域名仍在线，也不把演示账号描述为生产账号。
- 若修改了 Demo 数据，录制后重新运行 `npm run seed:demo` 恢复统一初始状态。
